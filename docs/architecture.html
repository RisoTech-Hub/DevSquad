Architecture & Data Flow

devsquad is built on a Feature-Centric Architecture (Vertical Slice). Instead of grouping by technical layers (e.g., all controllers together, all services together), we group by business capability.

Feature-Centric Layout

The codebase is split into two primary directories:

  • src/features/ — Contains self-contained modules: agent, config, daemon, doctor, github, project, slack. Each feature encapsulates its own domain, application, infra, and CLI commands.
  • src/shared/ — Contains cross-cutting concerns and shared infrastructure implementations: Redis, Tmux, Docker.

Data Flow: Slack to Sub-Agent

When a user sends a message in a project's Slack channel, it travels through several layers before an AI agent acts on it.

Message Flow Breakdown

  1. SlackListenerDaemon: Listens globally via Socket Mode. It determines the project context and pushes the event to a Redis queue.
  2. MessageProcessorDaemon: One per project. It pops messages from the project's specific Redis queue, formats them, and injects them into the corresponding tmux session using tmux send-keys.
  3. Claude Orchestrator: Running inside tmux, receives the text where the API Manager delegates to the specific sub-agents.

Shared Infrastructure

devsquad relies on several core infrastructure components:

Redis

Message queue backend. Each project has a dedicated queue key (e.g., queue:my-project) for reliable message delivery between the listener and processor daemons.

Tmux

Terminal multiplexer that hosts the Claude CLI sessions. Each project runs in its own named session, allowing multiple concurrent AI development teams.

Docker

Container runtime for agent isolation. Each agent can optionally run in its own Docker container for resource and environment isolation.

Note

The daemon system uses macOS launchd (LaunchAgents) to ensure persistence across reboots and automatic restart on failure.

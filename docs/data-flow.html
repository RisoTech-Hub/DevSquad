End-to-End Data Flow

Messages traverse multiple layers to ensure resilient and decoupled integration between Slack and the Claude orchestrator running in local tmux sessions.

1
Slack Workspace User posts a message to a project channel. Payload is transmitted over a Socket Mode WebSocket.
↓
2
SlackSocketModeAdapter Receives the WebSocket event. Filters out bot messages to avoid loops and monitors connection heartbeat.
↓
3
SlackListenerDaemon Matches the incoming channelId to a projectName binding map. Dispatches the JSON stringified message to a project-specific Redis queue (queue:{project}).
↓
4
Redis Queue Persists the message until it can be processed. Utilizes BRPOP blocking pop with a 5-second timeout.
↓
5
MessageProcessorDaemon Pulls the message from the Redis queue. Parses the JSON and formats the text: [Channel: #X | User: @Y | TS: Z | Thread: W] {text}.
↓
6
TmuxService.sendMessage() Executes terminal commands to safely inject the formatted message using tmux send-keys -l "{text}" {target}, sleeps 1 second, then hits Enter.
↓
7
Claude CLI The agent inside the local tmux session processes the command. If replying, it invokes the devsquad slack reply CLI tool natively.
↓
8
Slack Workspace (Response) The final response lands back in the Slack thread asynchronously.

Status Update Side-Channel

Independent of message traffic, daemons provide real-time updates:

  • TeamStatusService: Polls local Docker containers (e.g., claude-lead, gemini-manager) and pushes an ASCII table summary to the general status channel via Slack message editing.
  • ProjectStatusService: Tracks project phases (Planning, Delegating, etc.) and specific agent activity, updating the dedicated status message inside the project's Slack channel.

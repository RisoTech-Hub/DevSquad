Daemon & Background Processes

The daemon system is the backbone of devsquad, enabling reliable message routing between Slack and your local development environment. It uses macOS launchd (LaunchAgents) to ensure persistence across reboots and automatic restart on failure.

Daemon Types

SlackListenerDaemon

A single global process that:

  • Opens a WebSocket connection to Slack via Socket Mode
  • Receives all messages from all project channels
  • Identifies the target project based on the channel ID
  • Pushes the payload to Redis for processing

MessageProcessorDaemon

One process per project that:

  • Pops messages from the project's specific Redis queue
  • Formats the message for the tmux session
  • Injects the message using tmux send-keys

Daemon Management

Command Description
devsquad daemon start Start listener and all project processors
devsquad daemon start --name <project> Start processor for a specific project (also starts listener if needed)
devsquad daemon stop Stop all daemons
devsquad daemon stop --name <project> Stop processor for a specific project only
devsquad daemon restart Restart all daemons
devsquad daemon restart --name <project> Restart processor for a specific project
devsquad daemon status Show PID and load status for all daemons
devsquad daemon logs Tail listener logs
devsquad daemon logs --name <project> Tail processor logs for a specific project
devsquad daemon remove Remove all daemon plists

LaunchAgent Configuration

The LaunchDaemonManager generates .plist files in ~/Library/LaunchAgents/:

Listener Plist

com.devsquad.listener — Runs the WebSocket receiver (single global instance)

Processor Plist

com.devsquad.processor.{projectName} — Dedicated tmux injector instances (one per project)

Note

All plists set KeepAlive: false, giving the devsquad CLI absolute authority to manage daemon lifecycle rather than the OS unexpectedly reviving broken processes.

Troubleshooting

If messages stop flowing:

  1. Check daemon status: devsquad daemon status
  2. Review logs: devsquad daemon logs
  3. Run diagnostics: devsquad doctor
